
- criticalClients === undefined → `[]` (silence)
- criticalClients === 0 → category='clean', weight=40

- criticalClients >= 3 → category='attention', weight=50
- Seleção determinística via `seed % pool.length`

## Heurísticas Operacionais

| criticalClients | Categoria | Peso | Exemplo |
|-----------------|----------|------|---------|
| `undefined` | — (silence) | — | `[]` |
| `0` | clean | 40 | "Gestão em dia" |
| `1` | mild | 45 | "Alguns clientes merecem atenção" |
| `2` | mild | 45 | "Há acompanhamentos importantes hoje" |
| `3` | attention | 50 | "Há sinais importantes na carteira hoje" |
| `10` | attention | 50 | "A carteira exige atenção adicional" |

**Detalhamento:**
- `clean` (0 clientes): frases de tranquilidade operacional, peso baixo (40) — aparece após identity
- `mild` (1-2): frases de consciência suave, peso moderado (45) — reconhecimento sutil
- `attention` (3+): frases de atenção direcionada, peso alto (50) — maior prioridade contextual

O peso 40/45/50 posiciona fragments operacionais APÓS temporal (10) e ANTES de identity (20-50 conforme tipo).

## Validação Arquitetural

A implementação valida:

| Aspecto | Como valida |
|---------|-------------|
| Provider extensibility | Nova layer adiciona sem reescrever providers existentes |
| Fragment composition | Operational fragments mix com temporal + identity |
| Layer orchestration | Composer integra nova layer mantendo pipeline |
| Priority handling | Weights 40/45/50 vs existing weights |
| Silence behavior | Provider retorna `[]` quando contexto indisponível |
| Determinism | Seed-based selection preservado |
| Isolation | Provider sem UI logic, sem async, sem APIs |
